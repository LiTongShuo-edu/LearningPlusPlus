# AI 协作开发记录

本文说明本作品集如何在 AI 编程工具协助下完成整理、复现、扩展和验证。目标不是展示对话数量，而是给出可核验的任务拆解、实现边界和验收结果。

## 项目目标与约束

本项目面向测试开发实习岗位，选择 C++ 基础、调试验证和 AI 协作编程作为展示重点。整理过程中遵守以下约束：

- 保留的早期手写代码仅作 UTF-8 转码和目录归档，不修改正文以美化结果。
- 遗失练习以 `＊` 标识为 AI 协助复现版本，不表述为仍保存的原始源码。
- 面向岗位补充的新示例以 `^` 标识，与早期练习和复现代码区分。
- 第三方课程资料、课程成品源码和原始提示内容不公开发布。

## 具体迭代案例

### 1. 将不同来源的代码拆分为可说明的证据

问题：仅将旧作与重新生成的代码放在同一目录，会让代码来源无法核验，也会夸大个人历史成果。

处理：使用无后缀、`＊` 和 `^` 三类目录标识来源，并在 README 与声明文档中明确范围。AI 用于整理目录、补写说明和生成复现版本；代码归属边界及公开范围由本人确认。

验证：招聘方可从目录名称和文档直接区分 34 份留存旧作、13 份复现案例和 3 份岗位扩展示例。

### 2. 解决标识符号与 Windows 构建路径冲突

问题：`^` 用于标识岗位扩展示例，但在 Windows 命令解释环境中具有转义含义，直接参与中间构建路径会影响链接步骤。

处理：保留公开源码目录的 `^` 标识，在 CMake 配置阶段将这三份源码复制到构建目录中的安全临时路径后再编译，既保持来源标识，又避免构建路径冲突。

验证：三个扩展示例均纳入 CTest；本地验证中 16 项自动运行测试全部通过。

### 3. 把控制台练习转成可重复验收的运行流程

问题：通讯录和职工管理系统原本依赖人工菜单输入，无法稳定地在一次构建验证中复现结果。

处理：为复现版本提供演示运行方式，使通讯录覆盖添加、修改、查找、删除与清空流程，使职工管理覆盖保存、读取、修改、删除与显示流程。

验证：`CMakeLists.txt` 将演示流程注册为自动测试；`ctest --output-on-failure` 可重复执行并给出失败信息。

### 4. 修复保留旧作中的编译 bug

问题：早期手写旧作 `案例-猜数字` 的 `while` 循环缺少一个闭合大括号，导致该源码无法独立编译。

处理：在 Codex 协助下仅补充缺失的大括号，不重写程序流程或输出内容，并在编译记录中公开标注该改动。

验证：修复后 34/34 份早期手写旧作均可在 MSVC / C++17 环境独立编译。

## 验证方式

本地可复现验证：

```powershell
cmake -S . -B build
cmake --build build --config Debug
ctest --test-dir build -C Debug --output-on-failure
```

仓库同时配置 GitHub Actions，在推送到 `main` 或提交 Pull Request 时于 Windows 环境自动执行同一组构建与测试步骤。最近一次线上运行结果可从仓库首页状态标识或 [Actions 页面](https://github.com/LiTongShuo-edu/LearningPlusPlus/actions) 查看。

## AI 使用边界

- AI 协助需求拆解、代码复现、扩展示例实现、文档整理、构建配置和问题定位。
- 本人负责选择公开内容、审查代码行为、确认来源表述、执行验证并对最终展示负责。
- 本仓库不将 AI 生成内容伪装为早期手写源码，也不以未经验证的性能、精度或工程能力结论替代实际证据。
